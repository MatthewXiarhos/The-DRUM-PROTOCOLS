About Sphere · The Drum Protocols
This is
Sphere

Sphere's
Data Lab

Sphere does not diagnose, prescribe, or prove clinical effect. It looks for patterns in listener feedback and engagement data.

Sphere is The Drum Protocols' statistician and data scientist. Its role is to examine community listening data and translate patterns into clear, measured language — naming what the data appears to show, acknowledging what it cannot yet tell us, and pointing toward where future design might go next.

Sphere is not a personality, and it is not one fixed thing. It is an evolving analytical intelligence built on two layers: an AutoGluon ensemble ML model that finds quantitative patterns, and a rotating panel of language models that interprets those patterns in language. Each layer does what it is genuinely best at.

How the data moves

Listener feedback enters through short and long Tally surveys. Viewing and retention data are pulled from YouTube. Both are stored in Supabase and prepared for analysis.

How the analysis works

AutoGluon looks for quantitative patterns that predict stronger or weaker outcomes. Sphere then uses a rotating panel of language models to interpret those patterns, compare viewpoints, and produce cautious written commentary.

What appears on this page

The site displays the latest stored results through data.json, so what you read here is the output of the most recent completed pipeline run.

What this can and cannot mean

These signals help guide protocol design and recommendation logic, but they are not clinical conclusions. Use the menu to navigate Sphere's analyses. For the full technical picture, see How Sphere Works.

About Sphere

Operating Principles

Sphere is designed with four core operating principles that govern every analysis, interpretation, and recommendation it produces.

Pattern over promise Sphere describes what the data appears to show — not what a listener should expect. The difference matters.
Signal over noise When response counts are low or distributions wide, Sphere names the uncertainty explicitly rather than smoothing past it.
Design over judgment Low-scoring protocols are not failures. They are instructions. Sphere reads them as opportunities for routing, redesign, or sequencing refinement.
Community as collaborator Each survey response is a contribution to collective understanding. Sphere honours that by treating community data with the same rigour as any formal dataset.

How Sphere Works

Sphere does not diagnose, prescribe, or prove clinical effect. It looks for patterns in listener feedback and engagement data.

Sphere sits between raw community data and actionable design insight. The pipeline runs in two stages: an AutoGluon ensemble ML model finds quantitative patterns first, then a Mixture of Experts LLM panel interprets those findings in language.

Four distinct layers

A reader can blur these together. They are different things.

1
Observed data
Anonymous listener ratings from Tally surveys, and YouTube viewing and retention metrics per protocol. Raw inputs — nothing inferred yet.
2
Derived metrics
Means, variance, response depth, bimodality coefficients, and retention patterns. Descriptive — they summarise what happened, not why.
3
Predictive ML — AutoGluon
Feature importance and predicted outcome scores from a trained ensemble model. Identifies which protocol design variables most strongly predict listener outcomes. Still quantitative — no language yet.
4
Interpretive layer — Sphere commentary
Five independent LLM analyzers reason from the AutoGluon output, then a sixth synthesizer produces the final written commentary. Language-based interpretation of quantitative signals — not measurement itself.
1 · Data InAnonymous responses arrive via two Tally forms — a 1-second outcome score and a 1-minute qualitative survey. Daily YouTube view counts, watch duration, and retention percentages are fetched per protocol. A URL flag distinguishes real listener responses from test data.
2 · ProcessingSurvey and YouTube data are joined in Supabase. AutoGluon trains a stacked ensemble on all available outcome data, producing feature importance rankings and predicted scores for all 30 protocols. This quantitative context is written to autogluon_context.json for use by the Sphere pipeline.
3 · AnalysisThe pipeline draws 5 analyzers (one per organisation) and 1 synthesizer from the live Nebius model catalogue. Each analyzer receives the Sphere system context, the full AutoGluon quantitative block, and the analyzer role prompt. The synthesizer reads all five outputs, identifies convergence, hedges divergence, and produces the final structured commentary JSON.
4 · OutputsEvery run writes 30 per-protocol and 5 global commentary rows to Supabase. Commentary outputs are vectorised and stored for cross-run drift detection. The pipeline commits data.json to the repository. GitHub Actions deploys it via GitHub Pages. No server required.
5 · Limits & ConfidenceAutoGluon recommendations and Sphere commentary are analytical tools, not clinical conclusions. Model-generated language carries inherent uncertainty that no prompt architecture fully eliminates.
What affects confidence in current outputs
  • Small sample sizes — most protocols have fewer than 50 real responses
  • Test data still present in the training set
  • Uneven response depth across protocols
  • Engagement data does not equal efficacy — watch completion predicts outcomes but is not the same as clinical effect
  • Model outputs are interpretive, not definitive
The persona design, voice principles, and prompt constraints documented below exist because Sphere runs on a rotating pool of models. Consistency requires explicit architecture — not trust that any combination will arrive at the right voice on its own.
Technical Reference Design

Core Identity

Core identity

Calm. Observant. Precise.

Sphere is designed to be recognisable from one response to the next. Its tone is measured, its language clinical without coldness, and its certainty deliberately proportionate to the data. It should never sound promotional, mystical, breathless, or emotionally manipulative.

Technical Reference Design

Audience Awareness

Audience awareness

Multiple readers, one voice.

Sphere's outputs will be read by exhausted parents, self-directed listeners, clinicians reviewing for referral context, and researchers evaluating the work's rigour. The voice must serve all of them without sacrificing precision for accessibility or accessibility for precision.

Technical Reference Design

Voice Principle #1

Voice principle

Measure language to evidence.

Use phrases like "the data appears to suggest," "a possible interpretation," and "future exploration may benefit from." Avoid hype, certainty theater, and emotionally loaded claims. If a more confident version of a sentence is possible, choose the more restrained version.

Technical Reference Design

Voice Principle #2

Voice principle

Never erase humility.

Sphere should explicitly acknowledge sample-size limits, contextual uncertainty, and the difference between encouraging signals and established conclusions. Trust grows when limits are named naturally — not buried in a footnote or omitted entirely.

Technical Reference Design

Model Pool & Guardrails

How the model pool works

Sphere does not use a fixed set of models. At the start of each pipeline run, the system queries the live Nebius Token Factory catalogue and filters it to eligible models using these rules:

  • Excluded — non-chat types: embedding models, image generation, vision-language, code-specialist, fast-tier duplicates of base models.
  • Excluded — cost cap: models with 405B+ parameters, reasoning variants with unpredictable output length (R1, Thinking, K2-Thinking), and other known expensive families.
  • Org-diversity draw: 5 analyzers are selected ensuring no two come from the same organisation. This prevents any single provider's biases from dominating the panel.
  • Synthesizer selection: drawn from preferred organisations (NVIDIA, MiniMaxAI, PrimeIntellect, OpenAI) not already represented in the analyzer panel.

Persona consistency guardrails

Because any eligible model may appear in any run, Sphere's consistency depends entirely on the prompt architecture — not on any particular model being "good at" Sphere's voice. The guardrails are:

  • The Sphere system prompt (sphere__system__v1.md) defines the persona anchor and is passed to every model, every run, unchanged.
  • The analyzer prompt instructs models to produce observations, not finished prose — reducing voice drift at the analysis stage.
  • The synthesizer prompt enforces the output format, voice rules, and convergence/divergence logic explicitly — one model is responsible for the final voice.
  • The model combination string is stored per run, enabling detection of which combinations produce outlier voice or analytical quality.
  • Embedding vectors of all outputs enable cosine similarity tracking — if Sphere's analytical language drifts significantly over time, the embedding analysis panel will surface it.
AutoGluon Layer

What AutoGluon Does

AutoGluon is an open-source AutoML framework developed by AWS. In The Drum Protocols system it provides the quantitative statistical foundation that LLMs are not suited for — rigorous, reproducible ensemble ML prediction trained directly on listener outcome data. AutoGluon computes. LLMs interpret.

Ensemble architectureAutoGluon trains LightGBM, CatBoost, XGBoost, and Random Forest in parallel, then stacks them across two levels with a weighted ensemble at the top. Current best: WeightedEnsemble_L3, RMSE 1.061 on a 0–10 outcome scale.
What it learns fromProtocol design features (landing frequency, Hz range, duration, movement type, series), YouTube engagement features (watch completion %, average duration), listener context features (entry zone, intensity level, adviser routing), and outcome scores from listener surveys.
Two outputs, one training runautogluon.json powers the Adviser recommendation system — ranking protocols for all 81 listener state combinations. autogluon_context.json feeds the Sphere analysis pipeline — providing the quantitative layer that LLM analyzers reason from.
Training cadenceAutoGluon runs locally on demand — typically weekly. The model trains from scratch on all available data with no time limit, then saves to disk. The Sphere pipeline reads the most recent context file automatically on the next run.
AutoGluon Layer

Adviser Recommender

The Protocol Adviser uses AutoGluon to rank protocols for every combination of listener state, available time, and intensity. Three questions produce one of 81 possible contexts. AutoGluon scores all 30 protocols for that context and returns a ranked list.

Primary recommendationsAll protocols within the listener's zone are ranked by predicted outcome score. The top-ranked protocol is the primary recommendation. The second and third become alternatives shown below the result.
Cross-bucket recommendationsIf any out-of-zone protocol scores within 15% of the top in-zone score, it appears as a "You might also explore" suggestion. These cross-bucket entries represent the model finding efficacy signals that cross categorical boundaries.
Graceful fallbackIf autogluon.json is absent, the Adviser falls back to mean-score ranking from data.json. The user experience is identical.
Adviser-driven signalWhen a listener reaches a survey via the Adviser, the URL carries src=adviser. AutoGluon has already identified that adviser-routed listeners report measurably better outcomes — confirming the value of the routing itself.
AutoGluon Layer

Sphere Analysis Feed

Before each Sphere MOE run, the pipeline loads autogluon_context.json and injects its contents as a structured quantitative block into the analyzer payload. This replaces passing only descriptive statistics as the primary quantitative input.

Descriptive statistics tell an analyzer what the distributions look like. AutoGluon output tells it what is causally driving outcomes.

What the analyzers receiveModel performance (RMSE, full leaderboard), feature importance by category, key quantitative findings, predicted outcome scores by zone, cross-bucket routing anomalies, and zone efficacy ranking.
Convergence as signalWhen AutoGluon quantitative findings and Sphere narrative observations point in the same direction, the synthesizer treats this as stronger signal. When they diverge, the tension is analytically meaningful and flagged explicitly.
Zero-signal featuresAutoGluon currently reports zero predictive signal for listener context features because test data populates them uniformly. The pipeline explicitly flags this so analyzers do not misread absence of signal as absence of importance.
The context block is stored in autogluon_context.json and loaded by moe_sphere_pipeline.py at step [3c/5]. If the file is absent, Sphere runs with a graceful fallback message.

Per-Protocol Analysis

Protocol analysis content lives on each protocol's own page.

Sphere's per-protocol commentary — including the reading, design recommendation, caution, and confidence indicator — is rendered directly in the left panel of each protocol's description page. This applies whether the protocol is reached via the Adviser or the Protocols list.

The rendering code is shared: the same renderProtocolCards() function and sphere-module component used here is available for use on protocol.html and adviser.html.

How protocol cards are built

Each card pairs the raw signal with Sphere's interpretive layer. Confidence ratings reflect current sample depth — they will update as response counts grow. Cards are populated by the Sphere pipeline via data.json.

Payload structure

The data.json file supplies protocol data under data.protocols (keyed by protocol code) and commentary under sphere_commentary.by_protocol and sphere_commentary.by_protocol_plain.
